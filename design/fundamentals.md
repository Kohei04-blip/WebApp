# 前提知識

## 用語
  
  - データ
    - 保存したい情報そのもの
  
  - テーブル
    - 同じ種類のデータをまとめる箱
  
  - カラム
    - テーブルの項目

  - レコード
    - 1件分のデータ
   
  - CRUD
    - Creat:作る
    - Read:見る
    - Update:更新する
    - Delete:削除する
   
  - エンティティ
    - DBに保存する対象となるもの
    - 例）User,Node,Quiz
   
  - リレーション
    - データ同士の繋がり
    - 例）1つの分野に複数のノードがある
  
  - 主キー
    - データを一意に区別する
    - 例）node.id = 1, node.id = 2

  - 外部キー
    - 別のテーブルとの繋がりを表す項目
    - 例）node.subject_id
   

# UI設計・画面設計

## UI設計の重要原則

### 原則①：画面には必ず目的がある
- この画面でユーザーは何をするのかを明確にする
- 例：
  - ホーム：今日の復習を確認する
  - クイズ：問題を解く
  - ノード：知識を整理する

---

### 原則②：1画面1役割
- 1つの画面に役割を詰め込みすぎない
- 例：
  - 一覧：データを見る
  - 詳細：内容確認
  - 作成：入力する

---

### 原則③：操作を必ず書く
- ボタン・クリック・入力などを明確にする
- 例：
  - 「保存」ボタン
  - 「削除」ボタン
  - 「次へ」ボタン

---

### 原則④：操作の結果を書く
- 操作後に何が起きるかを定義する
- 例：
  - 保存 → DBに保存
  - 回答 → 正解表示
  - 評価 → 次回復習日更新

---

### 原則⑤：画面遷移を書く
- どの画面からどの画面へ移動するか明確にする
- 例：
  - ホーム → クイズ
  - クイズ → 次の問題
  - ノード → 詳細

---

### 原則⑥：データを意識する
- どんなデータが必要か考える
- 例：
  - ノード → title, content
  - クイズ → question, answer
  - 復習 → result, date

---

### 原則⑦：例外を考える
- エラーや空状態を考慮する
- 例：
  - 入力なし → エラー表示
  - データなし → 空メッセージ表示
  - クイズ0件 → ガイド表示

---

### まとめ
UI設計 = 操作とデータの設計

---


## UI設計の作成フロー

### STEP① 課題を1行で書く
- 例：
  - 復習の優先順位が分からず、学習が定着しない

---

### STEP② ユーザーの行動を書く
- 例：
  - 学習する
  - 記録する
  - 復習する
  - 定着する

---

### STEP③ 必要な画面を洗い出す
- 例：
  - ホーム
  - 復習一覧
  - クイズ
  - ノード一覧
  - ノード詳細
  - 作成画面

---

### STEP④ 各画面の目的を書く
| 画面 | 目的 |
|------|------|
| ホーム | 今日やること確認 |
| クイズ | 復習する |
| ノード | 知識整理 |

---

### STEP⑤ 画面の項目を書く
- 例（クイズ画面）：
  - 問題文
  - 回答入力欄
  - 提出ボタン
  - 正解表示
  - 理解度ボタン

---

### STEP⑥ 操作を書く
- 例：
  - 提出 → 回答判定
  - 理解度クリック → 次回復習日更新

---

### STEP⑦ 画面遷移を書く
- 例：
  - ホーム → クイズ
  - クイズ → 次の問題
  - ノード → 詳細

---

### STEP⑧ 必要なデータを洗い出す
- 例：
  - Node
  - Quiz
  - ReviewHistory

---

### STEP⑨ ロジックを書く
- 例：
  - 優先度 = 復習日超過 + 理解度低い + 未復習

---

### STEP⑩ MVPを決める
- 最初に作る最小機能に絞る
- 例：
  - ノード作成
  - クイズ作成
  - 復習機能

---

### まとめ
思考 → 設計 → 実装 の順で進める

---

  
## テーブル設計とは

- どんな種類の情報を、どんな項目で保存するかを決めること

- 例）Nodeテーブルを作る場合
  - id
  - title
  - content
  - subject_id
  - parent_node_id

## クラス設計とは

- プログラムの中でどんな役割の部品を作るかを決めること

1. データを表すクラス  
  例）
    - User
    - Subject
    - Node
    - Quiz
    - Review History

2. 処理を担当するクラス  
  例）
    - NodeService
    - QuizService
    - ReviwService
   

## API設計とは

- 画面からのリクエストを、サーバーにどう渡すかを決めること

  例えば、
  - ノード一覧を表示したい
  - ノードを新規登録したい
  - クイズ結果を保存したい
 
  こういう要求をどんな入り口で受けるかを決める

  入り口の例としては、

  - GET/nodes  
    ノード一覧を取得する
  - GET/nodes/{id}  
    ノード詳細を取得する
  - POST/nodes  
    ノードを新規作成する
  - POST/reviews/{quizId}/answer  
    クイズ回答結果を保存する
